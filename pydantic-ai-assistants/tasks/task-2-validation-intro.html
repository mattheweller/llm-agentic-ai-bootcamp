Introduction to Validation Libraries

Learning Objectives

  • Understand Pydantic's role in Python ecosystems
  • Learn Zod as the JavaScript equivalent
  • Compare syntax between Pydantic BaseModel and Zod objects
  • Create and test your first validation schema

Why Validation Matters

When building LLM applications, you need to ensure outputs match expected structures. Python developers use Pydantic, while JavaScript developers use Zod. Both provide runtime validation with type safety.

Learning Approach: We'll show Pydantic examples as reference, then implement working code with Zod.

Pydantic in Python (Reference)

Pydantic uses Python classes with type hints to define data models:

Python + Pydantic
from pydantic import BaseModel

class User(BaseModel):
    name: str
    age: int
    email: str

# Valid data - creates User instance
user = User(
    name="Mira",
    age=30,
    email="mira@gmail.com"
)
print(user.model_dump())
# {'name': 'Mira', 'age': 30, 'email': 'mira@gmail.com'}

# Invalid data - raises ValidationError
user = User(
    name="Mira",
    age="not-a-number",  # ❌ Wrong type
    email="mira@gmail.com"
)
# pydantic.ValidationError: 1 validation error for User
# age: Input should be a valid integer
JavaScript + Zod
import { z } from 'zod';

const UserSchema = z.object({
  name: z.string(),
  age: z.number().int(),
  email: z.string().email()
});

// Valid data - returns validated object
const user = UserSchema.parse({
  name: "Mira",
  age: 30,
  email: "mira@gmail.com"
});
console.log(user);
// { name: 'Mira', age: 30, email: 'mira@gmail.com' }

// Invalid data - throws ZodError
const user = UserSchema.parse({
  name: "Mira",
  age: "not-a-number",  // ❌ Wrong type
  email: "mira@gmail.com"
});
// ZodError: Expected number, received string

Key Syntax Differences

Feature Pydantic (Python) Zod (JavaScript)
Define Schema class User(BaseModel) z.object({ ... })
String Type name: str name: z.string()
Number Type age: int age: z.number().int()
Email Validation from pydantic import EmailStr
email: EmailStr		 email: z.string().email()
Array Type items: list[str] items: z.array(z.string())
Validate Data User(**data) UserSchema.parse(data)
Optional Field nickname: str | None nickname: z.string().optional()

Common Validation Patterns

Both libraries support rich validation beyond basic types:

Python + Pydantic
from pydantic import BaseModel, Field

class Product(BaseModel):
    name: str = Field(min_length=1, max_length=100)
    price: float = Field(gt=0)  # Greater than 0
    in_stock: bool
    tags: list[str] = []  # Default empty list
JavaScript + Zod
const ProductSchema = z.object({
  name: z.string().min(1).max(100),
  price: z.number().positive(),  // Greater than 0
  in_stock: z.boolean(),
  tags: z.array(z.string()).default([])  // Default empty
});

Why This Matters for LLMs

When you ask an LLM to return structured data, validation ensures you get exactly what you expect:

  • ✅ Type Safety: Numbers are numbers, strings are strings
  • ✅ Required Fields: No missing data that crashes your app
  • ✅ Format Validation: Emails are valid, URLs are well-formed
  • ✅ Clear Errors: Know exactly what's wrong when validation fails
