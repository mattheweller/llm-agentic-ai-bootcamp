Task 8

Evaluate the Fine-Tuned Model

Learning Objectives

  • Load and merge a LoRA adapter into the base model for inference
  • Run the same evaluation pipeline on the fine-tuned model
  • Compare before/after confusion matrices and per-class metrics
  • Interpret what the improvement means in a production context

Loading the Fine-Tuned Model

After training, the adapter weights live in the output directory. To evaluate, reload the base model and attach the adapter using PEFT's PeftModel.from_pretrained():

Python — Load Adapter for Inference 
from peft import PeftModel

adapter_path = "./sentiment_finetuned_adapter"

# Reload base model (quantized)
quant_config = BitsAndBytesConfig(load_in_4bit=True)
base_model = AutoModelForCausalLM.from_pretrained(
    "google/gemma-3-1b-it",
    quantization_config=quant_config,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained(adapter_path)

# Attach LoRA adapter
tuned_model = PeftModel.from_pretrained(base_model, adapter_path)
tuned_model.eval()

The adapter weights are automatically merged into the base model's forward pass. Inference uses the exact same function as the zero-shot baseline — only the model object changes.

Interpreting the Results

  • Neutral class improved most in precision (+0.31): The fine-tuned model stopped over-predicting neutral for ambiguous cases.
  • Negative class is still the hardest (F1 0.83 vs. 0.91 for neutral): The class is the smallest (12.5% of data) and shares vocabulary with neutral. More negative training examples would help.
  • Errors concentrate at neutral/positive boundary: 20 neutral headlines predicted as positive. These are likely genuine "soft positive" headlines that are ambiguous even to human annotators.

Production Considerations

When to retrain?

Monitor model accuracy on a holdout set monthly. If F1 drops below your SLA (e.g., 0.80), gather new labelled data and retrain the adapter — not the full model.

Serving the adapter

Deploy via vLLM or TGI. Load the base model once; swap adapters per-request using LoRA serving mode. Multiple fine-tuned tasks share one base model.

Data drift

Financial language evolves — "ESG" language, new product categories, regulatory terms. Periodic retraining on recent labelled data keeps the model calibrated.

Module Complete! You've built a full fine-tuning pipeline: dataset loading → formatting → baseline evaluation → LoRA training → results comparison. The pattern transfers to any classification or extraction task with labelled data.

Key Takeaways

  1. Fine-tuning outperforms prompting on narrow, domain-specific classification tasks
  2. LoRA trains <1% of model weights — making fine-tuning accessible on a single consumer GPU
  3. Always establish a zero-shot baseline before investing in fine-tuning
  4. F1-score is more informative than accuracy for imbalanced multi-class problems
  5. The adapter (4MB) embodies the entire task adaptation — easy to version and swap
